# 康小伴 AI 智能家居赋能养老院商业化闭环方案

## 1. 产品定位

康小伴定位为面向养老机构的 AIoT 智慧养护操作系统，以长者安全、养护执行、专业评估和机构经营为核心，不是单纯的智能家居平台，也不是自动诊疗系统。

产品连接护工端、养老医师端、管理端、智能设备、机构管理人员和家属，形成以下闭环：

```text
智能设备与人工记录
  -> 统一事件平台
  -> 确定性规则与 AI 风险分析
  -> 分级告警和任务
  -> 护工、养老医师、管理人员执行
  -> 执行证据与人工确认
  -> 风险闭环和质量评价
  -> 家属服务、机构报表和收费续约
  -> 反向优化规则、排班和个人养护计划
```

核心原则：

> 设备负责感知，规则负责安全，AI 负责整理和建议，工作人员负责确认，系统负责追踪闭环，管理端负责持续改进，商业系统负责收费和续约。

## 2. 业务闭环

### 2.1 感知层

可接入的养老院智能设备包括：

- 智能床垫：离床、睡眠、心率和呼吸趋势；
- 门磁和人体传感器：夜间活动、异常滞留；
- 床头、卫生间和随身紧急呼叫器；
- 毫米波跌倒雷达或本地视觉分析设备；
- 智能药盒：提醒、开盒、漏服和执行反馈；
- 血压计、血糖仪、血氧仪和体温计；
- 定位手环：电子围栏、走失预警和活动量；
- 烟感、燃气、水电和空气质量设备；
- 护工手机、平板、护理站等人工记录终端。

设备产生的是测量值、状态和事件，不能直接产生诊断结论。

### 2.2 判断层

后端将数据处理为四个层级：

1. 原始设备遥测数据；
2. 标准化健康、照护、环境和设备事件；
3. 确定性规则告警；
4. AI 趋势汇总、证据整理和任务建议。

跌倒、呼救、走失、长时间离床、设备安全等高风险事件必须由确定性规则或经过验证的专用模型触发，生成式 AI 不能成为唯一判断者。

### 2.3 执行层

事件根据职责自动分派：

- 护工：现场确认、养护任务、日常巡查和执行记录；
- 养老医师：综合评估、慢病与用药复核、风险确认和转介建议；
- 管理人员：人员补位、质量复盘、资源调度和设备维护；
- 康复师、营养师：专业计划和执行反馈；
- 家属：接收经过机构审核的服务摘要和风险通知。

任务至少包含创建、派发、接收、到场、处理、复核、关闭和升级状态，并记录时间、人员与证据。

### 2.4 复盘层

系统持续形成机构运营指标：

- 告警响应时间；
- 任务按时完成率；
- 风险事件重复发生率；
- 跌倒、压疮和走失事件率；
- 综合评估完成率；
- 用药核对完成率；
- 床位使用率；
- 人员负荷和排班缺口；
- 设备在线率和故障处理时间；
- 家属满意度。

指标反向调整规则、人员排班、养护计划和机构工作流程。

## 3. 后端总体架构

第一阶段建议采用模块化单体，而不是立即拆分大量微服务。业务和团队成熟后，再将 IoT、消息、AI 和分析服务按负载独立部署。

```text
HarmonyOS 护工端 / 养老医师端 / 管理端
                    |
              HTTPS / WebSocket
                    |
                API Gateway
                    |
    +---------------+----------------+
    | 身份、租户、角色与数据权限      |
    | 长者、房间、床位与入住管理      |
    | 综合评估和个人养护计划          |
    | 任务、巡查、交班和工单          |
    | 告警、风险事件和转介            |
    | 人员、班次和排班                |
    | 设备、物资和维护                |
    | 消息、家属服务和通知            |
    | 报表、质量指标和商业计费        |
    | AI 编排、知识库和审计           |
    +---------------+----------------+
                    |
 PostgreSQL / Redis / 对象存储 / 消息队列
                    |
              MQTT IoT Gateway
                    |
     厂商网关 / 边缘网关 / 智能设备
```

### 3.1 推荐技术栈

- 后端：Java Spring Boot 或 Kotlin；
- 关系数据库：PostgreSQL；
- 缓存和短期状态：Redis；
- 设备消息平台：EMQX/MQTT；
- 异步事件：RabbitMQ，规模扩大后再考虑 Kafka；
- 文件和报告：S3 兼容对象存储；
- 实时消息：WebSocket；
- 搜索：早期使用 PostgreSQL，规模扩大后使用 OpenSearch；
- 部署：Docker，早期不强制使用 Kubernetes；
- 可观测性：OpenTelemetry、Prometheus、Grafana 和 Loki。

## 4. 后端业务模块

### 4.1 租户、身份和权限

- 支持集团、机构和养护区域三级组织；
- 支持护工、养老医师、管理者、康复师、营养师、设备运维和家属角色；
- RBAC 只控制功能权限，另行控制机构、区域、长者和敏感字段的数据范围；
- 支持员工停用、离职撤权、会话失效和全操作审计；
- 登录页角色只能用于演示，生产系统角色必须由服务端授权决定。

### 4.2 长者与床位

- 长者基础档案、联系人和授权关系；
- 房间、床位、区域和入住状态；
- 入住、调床、短期离院、返回和退住；
- 照护等级、风险标签和服务套餐；
- 数据导入、合并、归档和历史追溯。

### 4.3 综合评估与养护计划

- ADL、认知、营养、跌倒、压疮和情绪等评估；
- 评估版本、有效期、执行人员和复核人员；
- 根据评估形成个人养护目标、项目、频率和责任团队；
- 计划变更需要保留版本和变更原因；
- AI 只能提供证据摘要和草稿，最终计划由专业人员确认。

### 4.4 任务与交班

- 周期任务、临时任务和事件触发任务；
- 按区域、班次、技能、工作量和优先级分派；
- 接单、转派、升级、退回和复核；
- 支持文字、数值、照片、设备记录和签名等执行证据；
- 自动生成交班清单，AI 可协助生成交班草稿。

### 4.5 告警与风险事件

- 接收设备、人工、规则和外部系统事件；
- 去重、合并、抑制、升级和关闭；
- 支持分级响应时限和超时升级；
- 记录现场确认、处置过程和复盘结论；
- 关联长者风险档案和养护计划调整。

### 4.6 人员和排班

- 员工、技能、证照和服务区域；
- 班次模板、排班、请假和临时补位；
- 基于长者数量、照护等级和任务量计算人员负荷；
- 识别人员缺口、连续工作和不合理班次；
- AI 可提供排班建议，但必须由管理者确认发布。

### 4.7 设备与物资

- 设备型号、厂商、序列号、协议和固件版本；
- 设备与机构、房间、床位或长者的绑定历史；
- 在线状态、电量、故障、校准和维护计划；
- 物资库存、领用、消耗、盘点和补货提醒；
- 建立认证设备目录，避免直接绑定某一个设备厂商。

### 4.8 家属服务

- 家属与长者关系认证和授权；
- 经过机构审核的养护动态、周报和服务记录；
- 风险通知分级和通知规则；
- 探访预约、服务评价和增值服务购买；
- 不向家属直接暴露内部未经确认的 AI 结论。

### 4.9 报表与商业计费

- 机构、区域、人员和长者维度的运营报表；
- 服务质量、风险闭环、人员负荷和设备指标；
- 套餐、订阅、床位数量、设备数量和 AI 调用额度；
- 合同周期、账单、续费、增购和欠费控制；
- 集团客户支持多机构汇总和分机构结算。

Handover                交接班

Telemetry               原始遥测数据

所有核心业务数据必须包含：

- `tenant_id`；
- `institution_id`；
- 创建人、创建时间、更新人和更新时间；
- 数据来源；
- 版本号；
- 有效、归档或删除状态；
- 必要的业务审计和幂等标识。

## 6. IoT 接入方案

### 6.1 统一设备模型

不同厂商的数据统一转换为：

```text
deviceId
deviceType
tenantId
institutionId
bindingTarget
metricCode
metricValue
unit
eventCode
eventTime
receivedTime
quality
firmwareVersion
rawMessageReference
```

### 6.2 设备安全

- 每台设备或网关使用独立身份；
- 使用 TLS、设备证书或短期凭证；
- 支持密钥轮换、禁用和吊销；
- 设备只能发布授权主题；
- 关键指令需要签名、幂等和回执；
- 原始数据保留周期与业务摘要分开管理。

### 6.3 边缘网关

养老院网络不稳定时，可部署边缘网关：

- 兼容 BLE、Zigbee、Wi-Fi、Matter 或厂商局域网协议；
- 网络中断时缓存事件；
- 本地执行呼救、跌倒和烟感等关键规则；
- 网络恢复后按顺序补传；
- 不在边缘设备中保存不必要的完整长者档案。

## 7. 典型事件闭环示例

以夜间离床为例：

1. 智能床垫检测长者离床；
2. MQTT 网关上报事件；
3. 后端验证设备身份、在线状态和长者绑定；
4. 规则引擎结合时间、持续时长和个人风险等级；
5. 生成二级风险告警；
6. 分派给所在养护区域值班护工；
7. 超过两分钟未接单，自动升级给班组长；
8. 护工到场确认原因并记录处置；
9. 若伴随跌倒或生命体征异常，升级养老医师；
10. 系统更新风险档案并提示复核养护计划；
11. 管理端统计响应时间和重复发生率；
12. 符合通知规则时，向家属发送经审核的摘要；
13. 风险复盘结果反向调整规则和夜间巡查计划。

## 8. AI 能力与安全边界

### 8.1 确定性规则层

负责安全强约束：

- 跌倒、呼救和走失；
- 长时间离床；
- 生命体征阈值和持续异常；
- 漏服和多次未执行；
- 烟感、燃气和设备离线；
- 任务超时和逐级升级。

规则需要版本、适用范围、启停状态、测试记录和发布审计。

### 8.2 预测模型层

可逐步建设：

- 跌倒风险变化；
- 压疮风险；
- 营养风险；
- 夜间异常活动；
- 风险再次发生概率；
- 人员工作量和排班需求。

模型输出必须展示输入因素、模型版本、时间范围、置信度和适用限制，并允许人工否决。

### 8.3 生成式 AI 层

适合处理：

- 汇总长者近期变化；
- 生成交班记录草稿；
- 整理综合评估缺失项；
- 生成家属周报草稿；
- 查询机构制度和操作流程；
- 汇总机构质量问题；
- 向管理者解释指标变化。

生成式 AI 不得自动完成最终诊断、医嘱、用药调整、转介决定、考核处分或向家属发布未经审核的信息。

### 8.4 AI 后端组件

```text
AI Gateway
  -> 权限与数据范围检查
  -> 敏感字段脱敏
  -> 上下文检索
  -> 模型路由
  -> 输出结构化校验
  -> 安全规则检查
  -> 人工确认
  -> 结果、模型版本和引用证据审计
```

- 异常处置和转介；

### 9.3 管理端

- 养护质量；
- 设备在线率和维护；
- 物资库存；
- 家属满意度；
- 经营、集团和监管报表。

## 10. 商业化模式

采用软件订阅、设备接入、实施服务和增值服务组合收费。

### 10.1 基础版

适合小型养老机构：

- 长者档案；
- 任务与交班；
- 基础告警；
- 护工端和管理端；
- 基础报表。

建议定价：每床每月 20～40 元。

### 10.2 专业版

增加：

- 养老医师端；
- 综合评估；
- 用药安全；
- 智能设备平台；
- 风险闭环；
- 家属周报；
- AI 养护助手。

建议定价：每床每月 60～100 元。

### 10.3 集团版

增加：

- 多机构管理；
- 集团数据驾驶舱；
- 私有化或专属云；
- 单点登录；
- 数据接口；
- 定制监管报表；
- 模型和知识库管理。

建议定价：年度项目制，20～100 万元起，按机构数量、床位、服务级别和部署方式报价。

### 10.4 其他收入

- 设备销售或租赁；
- 每台设备接入服务费；
- 实施和数据迁移费；
- 私有化部署费；
- 运维服务费；
- AI 调用包；
- 家属增值会员；
- 保险、康复、营养和远程服务合作分成。

硬件不建议全部自研生产。应建立认证设备目录，通过协议适配、项目实施、租赁和渠道利润获得收入。

## 11. 销售与客户成功闭环

销售重点不是“使用了 AI”，而是可量化结果：

- 降低遗漏和重复登记；
- 缩短告警响应时间；
- 提高任务完成率；
- 减少风险重复发生；
- 提高床位和人员利用率；
- 提升家属信任；
- 降低检查、审计和报表准备成本。

推荐试点路径：

1. 选择一家 80～200 床机构；
2. 首期覆盖一个养护区和 20～40 位长者；
3. 接入呼叫、离床和生命体征三类设备；
4. 运行 8～12 周；
5. 对比上线前后的响应时间、漏项率和任务完成率；
6. 形成客户案例和投资回报报告；
7. 扩展到全院并签订年度订阅；
8. 复制到同集团其他机构；
9. 持续通过运营报告推动续费和模块增购。

## 13. 安全与合规底座

养老业务涉及身份、健康、位置和家庭关系数据，必须具备：

- 多租户强隔离；
- 最小权限和数据范围控制；
- 传输与存储加密；
- 敏感字段脱敏；
- 全操作审计；
- 设备身份和密钥轮换；
- AI 输入脱敏和输出审计；
- 数据导出审批；
- 备份、恢复和容灾；
- 员工离职即时撤权；
- 家属授权和可见范围控制；
- 数据保存期限和销毁策略；
- 开发、测试和生产环境隔离。

## 14. 首批后端交付物

正式开发前，应先完成以下设计，而不是直接按前端 Mock 页面拼接接口：

1. 领域模型和业务边界；
2. 数据库 ER 图和租户隔离策略；
3. 设备事件标准和 MQTT Topic 规范；
4. 告警分级、去重和升级规则；
5. 任务状态机；
6. OpenAPI 接口契约；
7. WebSocket 消息契约；
8. RBAC 和数据范围矩阵；
9. 审计、日志和敏感字段规范；
10. AI 输入、输出、引用证据和人工确认协议；
11. 套餐、订阅和计费模型；
12. 试点机构验收指标。

## 15. 商业成功指标

### 产品指标

- 日活跃工作人员比例；
- 任务数字化覆盖率；
- 告警有效率；
- 平均响应时间；
- 任务按时完成率；
- AI 建议采纳率和人工修改率；
- 设备在线率。

### 客户价值指标

- 漏项率下降比例；
- 风险重复发生率变化；
- 人均记录时间减少量；
- 报表准备时间减少量；
- 家属满意度变化；
- 床位使用率和续住率变化。

### 商业指标

- 每床月收入；
- 单机构年度合同额；
- 软件毛利率；
- 设备接入毛利率；
- 客户获取成本；
- 上线周期；
- 模块增购率；
- 年度续费率；
- 集团机构复制率。

## 16. 最终结论

康小伴的商业壁垒不能只建立在前端页面或通用大模型上，而应建立在以下组合能力上：

- 养老机构真实工作流；
- 多角色任务协同；
- 统一设备接入；
- 可配置的安全规则；
- 可追溯的风险闭环；
- 专业人员确认的 AI 辅助；
- 可量化的质量与经营指标；
- 可复制的实施、收费和续约体系。

后端建设应首先打通“设备事件—规则告警—人员任务—执行证据—风险复盘—经营指标”主链路，再逐步增加预测模型、集团管理和商业计费能力。
